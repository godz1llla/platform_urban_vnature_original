# URBAN COLLEGE PLATFORM — Техническая документация

## 1. Обзор системы
**URBAN COLLEGE PLATFORM** — это комплексная веб-система для управления образовательным процессом в колледже. Система автоматизирует учет посещаемости (через QR-коды), ведение журнала оценок, управление расписанием и льготным питанием в столовой.

---

## 2. Роли и функции (Матрица доступа)

В системе реализовано 5 основных ролей. Ниже описано, кто и какие функции выполняет.

### 🔐 Администратор (Admin)
*Главный управляющий системой. Имеет доступ ко всем данным.*
- **Управление пользователями**: Создание, редактирование и удаление аккаунтов (студенты, преподаватели, персонал).
- **Управление структурой**: Создание специальностей, учебных групп и учебных предметов.
- **Расписание**: Полный контроль над сеткой занятий для всех групп и преподавателей.
- **Столовая**: Настройка категорий питания, установка лимитов (порций), назначение студентов на льготы.
- **Статистика**: Просмотр глобальных отчетов по посещаемости и успеваемости.

### 👨‍🏫 Преподаватель (Instructor)
*Обеспечивает учебный процесс.*
- **Посещаемость**: Генерация динамического QR-кода на занятии. Просмотр списка студентов, которые отметились (сканировали код).
- **Журнал оценок**: Выставление баллов студентам по своим предметам.
- **Расписание**: Просмотр своего графика занятий с указанием групп и аудиторий.
- **Статус**: Видит текущие и предстоящие пары в реальном времени.

### 🎓 Студент (Student)
*Основной пользователь контента.*
- **Посещаемость**: Сканирование QR-кода преподавателя для отметки присутствия на паре.
- **Успеваемость**: Просмотр своих оценок и процента посещаемости в личном кабинете.
- **Расписание**: Актуальный график занятий своей группы на неделю.
- **Столовая**: Просмотр своего баланса (лимитов) питания и истории получения порций.

### 📋 Куратор (Curator)
*Осуществляет надзор за вверенными группами.*
- **Мониторинг**: Просмотр списка студентов своих групп.
- **Отчетность**: Доступ к детальному ежедневному отчету (кто был на какой паре).
- **Пропуски**: Просмотр списка отсутствующих студентов за сегодня.
- **Успеваемость**: Анализ успеваемости студентов своей группы.

### 🍴 Оператор столовой (Cafeteria Operator)
*Обеспечивает выдачу питания.*
- **Выдача**: Регистрация порции (списание лимита) при сканировании QR-кода студента или показе своего кода оператора.
- **Проверка**: Моментальная проверка прав студента на бесплатное питание.
- **История**: Просмотр списка выданных порций за день.

## 3. Как работают основные процессы

### 📅 Система посещаемости (QR Attendance)
1. **Преподаватель** нажимает "Начать отметку" -> генерируется QR-код.
2. QR-код является **динамическим** (обновляется каждые 8 секунд) для защиты от пересылки скриншотов.
3. **Студент** в приложении сканирует код -> данные (ID студента, ID преподавателя, время) уходят на сервер.
4. Сервер проверяет валидность токена и создает запись в `attendance_records`.

### 📝 Журнал оценок (Grades)
1. Основные оценки выставляет **Преподаватель** через интерфейс журнала.
2. Данные сохраняются в таблицу `grades` с привязкой к предмету и дате.
3. **Администратор** может корректировать любые оценки в случае ошибок.

### 🍴 Льготное питание (Cafeteria Module)
1. **Админ** создает категорию (например, "Сироты") и задает лимиты (например, 1 раз в день).
2. Студенту назначается категория.
3. При получении еды происходит транзакция. Система автоматически проверяет, не превышен ли суточный/недельный/месячный лимит.

## 4. Техническая архитектура

- **Backend**: Чистый PHP 8.3 (без тяжелых фреймворков для максимальной скорости).
- **База данных**: MySQL 8.0.
- **Frontend**: Ванильный JavaScript (ES6+), HTML5, CSS3.
- **Авторизация**: JWT (JSON Web Tokens). Токен хранится в `localStorage` и передается в заголовке `Authorization: Bearer <token>`.
- **API**: REST-подобный интерфейс, маршрутизация через `backend/api.php`.

---

## 5. Что должно быть в проекте "правильно" (Идеальное состояние)

1. **Безопасность**: Все пароли в БД должны быть хешированы (`password_hash`). Доступ к API — только по валидному JWT.
2. **Целостность данных**: Связи между таблицами (Foreign Keys) должны обеспечивать каскадное поведение или запрет удаления (например, нельзя удалить группу, в которой есть студенты).
3. **Адаптивность**: Интерфейс студента должен быть Mobile-First (удобен для смартфона).
4. **Масштабируемость**: Код разбит на контроллеры (Controllers) по функциональным модулям.
